We need to implement the first-time setup foundation for lxp-llm-gateway.

Important: do not implement the full frontend wizard yet. Start with the backend foundation and CLI/env bootstrap design.

Context:
lxp-llm-gateway is a self-hosted, multi-tenant LLM gateway. The first install experience must feel like a serious self-hosted product.

Architecture decision:
- CLI prepares technical environment.
- Setup wizard configures application/domain data.
- Technical secrets must come from env generated by the CLI.
- The wizard must never generate or store JWT/encryption secrets.
- Provider API keys must be stored using the existing encrypted credential storage.
- Setup endpoints must be disabled permanently after installation is completed.

Security requirements:
- Setup is available only if no super admin exists and installation_state is not COMPLETED.
- Setup endpoints must require X-Setup-Token.
- The setup token should be generated by the CLI.
- Prefer storing SETUP_TOKEN_HASH in env instead of raw SETUP_TOKEN.
- Never log setup token, provider API keys, JWT secrets, encryption keys, or generated integration API keys.
- Setup bootstrap must be atomic and transactional.
- Setup must not leave the app half-configured.

Backend tasks:
1. Inspect existing auth/user/tenant/provider credential architecture.
2. Add InstallationState persistence:
  - status: PENDING | IN_PROGRESS | COMPLETED
  - setupStartedAt
  - setupCompletedAt
  - completedByUserId
  - appVersion if available
  - createdAt
  - updatedAt
3. Add a safe setup status endpoint:
   GET /api/v1/setup/status
   It should expose only safe booleans:
  - setupRequired
  - setupCompleted
  - tokenRequired
  - version if available
4. Add setup token guard/middleware for setup endpoints.
5. Add provider credential test endpoint:
   POST /api/v1/setup/providers/test
   It should accept provider + credential temporarily, test it, and never persist/log the secret.
6. Add final bootstrap endpoint:
   POST /api/v1/setup/bootstrap
   It should:
  - verify setup is allowed
  - verify setup token
  - create first super admin
  - create first tenant
  - create initial tenant membership
  - configure initial policies
  - store provider credentials using encrypted storage
  - optionally create OpenWebUI integration client/API key
  - mark installation completed
  - return generated integration API key once if created
7. Add tests:
  - setup status when fresh install
  - setup status when completed
  - bootstrap rejected without token
  - bootstrap rejected with invalid token
  - bootstrap succeeds once
  - bootstrap cannot run twice
  - provider keys are not logged
  - installation_state is completed after successful bootstrap
  - rollback if one bootstrap step fails

CLI tasks:
1. Add a TypeScript CLI command for initial env generation.
2. Use a pleasant terminal UX with prompts, colors, and arrow selection.
3. Generate secure random values for:
  - JWT secrets
  - credential encryption key
  - setup token
4. Store only setup token hash in .env if practical.
5. Do not overwrite existing .env without confirmation.
6. Print:
  - setup URL
  - setup token once
  - next docker compose command
7. Add a doctor command that validates required env vars.

Do not start by making broad unrelated refactors.
Follow existing project conventions.
Before editing files, inspect the repo structure and propose the implementation plan.
